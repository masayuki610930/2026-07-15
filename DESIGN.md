# 懇親会用 MBTI診断 Web — 設計書

## 1. 目的・前提

- 会社組織の懇親会（参加者 約50人）で使う、コミュニケーション促進のための簡易MBTI診断。
- 12問の質問に答えると16タイプのいずれかが表示される。
- 厳密な心理測定ではなく、会話のきっかけ作りが目的の「エンタメ診断」。
- 想定アクセス環境: 大半が **iPhone（Safari）**。モバイルファースト。
- 配布方法: **GitHub Pages** での静的公開。ビルド不要の **単一HTMLファイル**（`index.html`）。
- 言語: 日本語のみ。

## 2. MBTIの基本（リサーチ要点）

一般的なMBTIは次の4つの二分軸（dichotomy）から成り、各軸で優勢な極を選んで4文字のタイプ（16種類）を決める。

| 軸 | 意味 |
|---|---|
| **E** Extraversion / **I** Introversion | 関心・エネルギーの向く方向（外向 or 内向） |
| **S** Sensing / **N** Intuition | 情報の取り入れ方（具体・現実 or 抽象・可能性） |
| **T** Thinking / **F** Feeling | 意思決定の仕方（論理 or 感情・関係性） |
| **J** Judging / **P** Perceiving | 外界への接し方（計画的 or 柔軟・即興的） |

参考: [Simply Psychology — MBTI](https://www.simplypsychology.org/the-myers-briggs-type-indicator.html), [SIGMA Assessment Systems — What is the MBTI](https://www.sigmaassessmentsystems.com/blog/myers-briggs/)

公式診断はどちらの極が優勢かを多数の設問への回答スコアから判定する。本コンテンツもこれに準拠し、シンプルな**多数決方式**を採用する。

## 3. 診断ロジック

- 12問 = **4軸 × 3問**。各問は **A/B の2択**で、A・Bそれぞれが軸の片方の極に対応。
- 軸ごとに3問中どちらが多く選ばれたかで極を決定（3問なので必ず2:1以上に分かれ、同点は発生しない）。
- 4軸の判定結果を連結して4文字のタイプコード（例: `ENFP`）を確定。
- 出題順は軸をブロックにせず、E/I→S/N→T/F→J/P の4軸をラウンドロビンで混ぜて提示し、「診断っぽさ」を抑える（採点には影響しない）。

# 拡張設計: 会場集計ダッシュボード（§11〜§17）

以降は既存の診断（§1〜§10）への追加機能の設計。**この設計書を正として実装する**（実装は別セッション）。

## 11. 拡張の目的・全体構成

- 参加者が診断結果を匿名で送信し、**会場全体のタイプ分布**を別画面でリアルタイムに眺められるようにする。
- 集計画面は回答画面と**URLを分ける**（別ファイル `dashboard.html`）。プロジェクタや司会者PCでの常時表示を想定。
- 集計バックエンドには **Google スプレッドシート + Google Apps Script (GAS) Web App** を使う。GitHub Pages は静的ホスティングのため、シートへの読み書きはすべて GAS 経由の HTTP API で行う。

```
[参加者 iPhone: index.html]
    │ POST {type, uuid}（結果画面の送信ボタン）
    ▼
[GAS Web App]───append──▶[Google スプレッドシート（非公開）]
    ▲ GET ?action=summary&key=READ_TOKEN（10秒ごと）
    │
[司会者PC/プロジェクタ: dashboard.html?key=XXXX]
```

決定事項（発注者確認済み）:

1. 送信データは**タイプのみ・匿名**（4文字コード＋タイムスタンプ＋冪等性用UUID。個人情報なし）。
2. 集計画面の閲覧は **URLトークン方式**（`?key=` をGASが検証。トークンはリポジトリに置かない）。
3. 送信は**端末ごとに1回まで**（localStorage フラグで制御）。
4. タイルには**会場の人数(%)と日本・世界の割合%を併記**（日本の分布は§15.4のリサーチ結果を採用）。

## 12. バックエンド設計（スプレッドシート + GAS）

### 12.1 スプレッドシート

- 発注者のGoogleアカウントで新規作成。**共有設定は変更しない（非公開のまま）**。アクセスは常にGAS経由。
- シート名 `responses`。1行目はヘッダー、以降1回答=1行。

| 列 | 内容 | 例 |
|---|---|---|
| A: `timestamp` | GASが付与するISO 8601時刻 | `2026-07-15T18:03:21.000Z` |
| B: `type` | 4文字タイプコード | `ENFP` |
| C: `uuid` | クライアント生成のUUID v4（冪等キー） | `9b2f...-...` |

集計はdoGet時に全行を数える（50人規模＋行数上限ありなので毎回全走査で十分。集計用の別シートや事前計算は作らない）。

### 12.2 GAS Web App（`gas/Code.gs`）

- リポジトリの `gas/Code.gs` を**正本**とし、Apps Script エディタへ手動で貼り付けてデプロイする（デプロイ手順はREADME）。
- デプロイ設定: 種類「ウェブアプリ」、**次のユーザーとして実行: 自分**、**アクセスできるユーザー: 全員**。
- 秘密情報の置き場所: **Script Properties**（`READ_TOKEN` = 集計閲覧用トークン）。コード・リポジトリには書かない。

#### API契約

**POST（回答送信）** — リクエストは `Content-Type: text/plain` のボディにJSON文字列を入れる。
（GASはCORSプリフライト(OPTIONS)に応答できないため、プリフライトが発生しない simple request にするための定石。`fetch(url, {method:"POST", body: JSON.stringify(...)})` でヘッダー指定なしにすれば `text/plain` になる）

```
リクエストbody: {"type":"ENFP","uuid":"<UUID v4>"}
レスポンス(JSON):
  成功       {"ok":true}
  重複UUID   {"ok":true,"duplicate":true}   ← 追記せず成功扱い（クライアントは区別不要）
  拒否       {"ok":false,"error":"invalid_type" | "invalid_uuid" | "invalid_body" | "capacity_exceeded"}
```

**GET（集計取得）** — `<エンドポイント>?action=summary&key=<READ_TOKEN>`

```
成功: {"ok":true,"total":37,"counts":{"INTJ":2,"INTP":1,...16タイプ全キー(0も含む)...},"updatedAt":"<ISO 8601>"}
拒否: {"ok":false,"error":"unauthorized"}   ← countsは絶対に含めない
```

- レスポンスは常に `ContentService.createTextOutput(JSON.stringify(...)).setMimeType(ContentService.MimeType.JSON)` で返す（GASのJSONレスポンスはリダイレクト先の googleusercontent.com から `Access-Control-Allow-Origin: *` 付きで配信されるため、クロスオリジンfetchで読める）。

#### doPost処理仕様（この順で実装）

1. `e.postData.contents` の長さが 500 文字を超えていたら `invalid_body` で拒否（巨大ボディ対策）。
2. `JSON.parse` する。失敗したら `invalid_body`。
3. `type` が16タイプコードの**ホワイトリストに完全一致**しなければ `invalid_type`（大文字化などの補正はしない）。
4. `uuid` が UUID形式の正規表現 `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i` に一致しなければ `invalid_uuid`。
5. `LockService.getScriptLock()` を取得（最大10秒待ち。取れなければ `invalid_body` 相当のエラーでよい）。**以降の重複チェック〜appendを排他区間にする**（同時送信でのUUID重複すり抜け・行数レース防止）。
6. C列（uuid）に同じ値が既にあれば追記せず `{"ok":true,"duplicate":true}`。
7. データ行数が **1000行** に達していたら `capacity_exceeded` で拒否（荒らしでシートが際限なく膨らむのを防ぐ上限。50人想定に対し20倍のマージン）。
8. `[new Date().toISOString(), type, uuid]` を append し `{"ok":true}`。
9. finally でロック解放。

#### doGet処理仕様

1. `e.parameter.action` が `summary` でない、または `e.parameter.key` が Script Properties の `READ_TOKEN` と**完全一致しない**場合は `unauthorized`。
2. 一致したら `responses` の全データ行を読み、タイプ別に集計。**16タイプ全キーを必ず含める**（0件も `0` で返す。クライアント側の欠損処理を不要にするため）。ホワイトリスト外の値が万一混入していても無視する。
3. `total`（有効行数）、`updatedAt`（現在時刻）と共に返す。

## 13. セキュリティ方針と限界

GitHub Pagesのソースは全公開であり、書き込みエンドポイントのURLは隠せない。その前提で担保すること:

| 担保する点 | 手段 |
|---|---|
| シート本体への直接アクセス不可 | シートは非公開のまま。認証情報・シートIDはGAS内部のみ（HTMLには一切置かない） |
| 不正データが入らない | doPostの厳格な検証（16タイプwhitelist・UUID形式・ボディ500文字上限） |
| リトライ・連打で水増しされない | UUID冪等（同一UUIDは1行のみ）＋クライアント側送信済みフラグの二重防御 |
| 荒らされても被害が有限 | 行数上限1000でそれ以上は受け付けない |
| 集計結果を部外者に見せない | doGetは `READ_TOKEN` 必須。トークンは**リポジトリ・HTMLに埋め込まず**、Script Propertiesに保存し、`?key=`付きフルURLを司会者にのみ共有 |

**明示する限界**（README・運用者向け）: 悪意ある参加者がソースからエンドポイントを特定し、UUIDを変えて偽タイプを複数送信することは技術的に防げない（認証なしの匿名投票の原理的限界）。懇親会のエンタメ用途では許容し、万一荒れた場合はシートの該当行を手で削除 → 次のポーリングで集計画面に反映される、という運用で対処する。

## 14. 回答画面（index.html）への追加

### UI

- 結果画面の「もう一回診断する」ボタンの**上**に送信ボタンを追加する。`result-actions` 内に配置し、既存の `.btn` スタイル系を踏襲:
  - ボタン文言: **「📊 会場の集計に参加する（匿名）」**
  - ボタン直下に補足文（`.share-note` 相当の小さい文字）: 「送信されるのはタイプ（4文字）だけ。名前などは送られません。」
- ボタンの状態遷移:

| 状態 | 表示 | 操作 |
|---|---|---|
| 未送信 | 📊 会場の集計に参加する（匿名） | タップで送信 |
| 送信中 | 送信中… | disabled |
| 成功 | 送信しました ✓ | disabled（以後この端末では常にこの表示） |
| 失敗 | 送信に失敗しました。もう一度タップ | 再タップでリトライ可 |

### ロジック

- `<script>` 冒頭に設定定数を置く:
  ```js
  var CONFIG = { endpoint: "" }; // GAS Web AppのURL（.../exec）。デプロイ後に設定
  ```
  `endpoint` が空文字のときは送信ボタン自体を**非表示**にする（バックエンド未設定でも診断単体で従来通り動く後方互換）。
- localStorage キー `mbti_submitted_v1` に `{"uuid":"...","sentAt":"..."}` を保存:
  - 初回送信時に `crypto.randomUUID()` でUUIDを生成し、**送信前に**localStorageへ保存（送信後のstorage失敗でUUIDを失わないため）。リトライ時は同じUUIDを使い回す → GAS側冪等と合わせて二重カウントされない。
  - `sentAt` が入っていれば送信済み。結果画面を出すたびに判定し「送信しました ✓」表示（**端末ごとに1回**。再診断しても再送不可）。
  - `crypto.randomUUID` が使えない環境（http配信のローカル確認等）は `Math.random` ベースのUUID v4フォールバックを実装する。
- 送信は `fetch(CONFIG.endpoint, { method:"POST", body: JSON.stringify({type:code, uuid:uuid}) })`。**ヘッダーは指定しない**（text/plainになりプリフライト回避）。レスポンスの `ok:true` を確認して成功扱い。ネットワークエラー・`ok:false`・タイムアウトは失敗表示。

## 15. 集計画面（dashboard.html・新規）

単一HTMLファイル（CSS/JS内包・外部ライブラリなし）。§7のビジュアル言語（ホログラフィック背景 `.bg`、グラスモーフィズムカード、虹色グラデーション見出し）を流用するが、**PC/プロジェクタ想定**のためコンテンツ幅は最大 **1100px**。タイルは4列グリッド、900px未満で2列に落とす。iPhoneでの閲覧も崩れないこと。

※ 単一ファイル方針のため、タイプのニックネーム・キャッチは `index.html` の `TYPES` から必要分（`nickname`/`catch`）をこのファイルにも複製して埋め込む（共有JSは作らない）。

### 画面構成（上から）

1. **ヘッダー**: タイトル「みんなの診断結果」＋合計回答数「回答: **37**人」＋最終更新時刻（`HH:MM:SS 更新`）＋接続ステータス（正常時は非表示、異常時のみ「⚠ 更新できません」を控えめに表示）。
2. **棒グラフレース**（カード内）: 16タイプ**全件**の横棒を回答数の**降順**に常時ソートして表示。
3. **タイルグリッド**（参照画像準拠）: 16タイプの**固定順** 4×4。

### 棒グラフレースの仕様

- 各行の構成: `[タイプコード＋ニックネーム] [横棒] [n人]`。
- 実装方式（ライブラリ不要のFLIP相当）:
  - コンテナは `position:relative`、高さ = 行高 × 16 で固定。
  - 各行は `position:absolute; left:0; right:0;` とし、順位に応じて `transform: translateY(順位 × 行高)` を設定。
  - 棒は `width: (count / maxCount * 100)%`（maxCount=0のときは全棒width 0）。
  - `transform` と `width` に `transition: 0.8s ease` を付けると、データ更新時に**行が上下にスライドし棒が伸縮する**「レース」の動きになる。
  - 同数のタイプはタイプコードのアルファベット順で順位を安定させる（更新のたびに同数同士が入れ替わってチラつくのを防ぐ）。
  - 値が増えた行は一瞬ハイライト（背景色を0.8s程度でフェードアウト）。
- 棒の色は4グループで色分け（タイルのアクセントと共通）:

| グループ | タイプ | 色 |
|---|---|---|
| 分析家 | INTJ / INTP / ENTJ / ENTP | パープル `#9b6dff` |
| 外交官 | INFJ / INFP / ENFJ / ENFP | グリーン `#34c39a` |
| 番人 | ISTJ / ISFJ / ESTJ / ESFJ | ブルー `#5b8def` |
| 探検家 | ISTP / ISFP / ESTP / ESFP | オレンジ `#ff9f43` |

### タイルグリッドの仕様

- 並びは参照画像と同じ固定順（行ごと）: `ISTJ ISFJ INFJ INTJ / ISTP ISFP INFP INTP / ESTP ESFP ENFP ENTP / ESTJ ESFJ ENFJ ENTJ`。
- 各タイル（白ベースのカード、下辺にグループ色2〜3pxのアクセントボーダー）:
  - 絵文字アイコン＋**タイプコード**（太字）＋ニックネーム（`TYPES.nickname`）
  - 一言キャッチ（`TYPES.catch`）
  - **会場: n人 (x%)** … スプレッドシート集計値。xは total に対する割合（四捨五入で小数1桁、total=0 のときは「会場: 0人」のみ）
  - 日本: y% ／ 世界: z% … §15.4の静的データ（HTMLに埋め込み。1行にまとめて小さめに表示）
- 会場0人のタイルは会場行をグレー表示。人数が**増えた**タイルは一瞬ハイライト（レースと同じ演出）。
- タイル用絵文字（参照画像のモチーフに寄せる）: ISTJ ⚖️ / ISFJ 🛡️ / INFJ 💡 / INTJ 👑 / ISTP 🔧 / ISFP 🎨 / INFP 💗 / INTP 🧪 / ESTP 🚀 / ESFP 🎤 / ENFP 📣 / ENTP 💬 / ESTJ 🏢 / ESFJ 🤝 / ENFJ 👥 / ENTJ 🏁

| タイプ | 日本% | 世界% | タイプ | 日本% | 世界% | タイプ | 日本% | 世界% | タイプ | 日本% | 世界% |
|---|---|---|---|---|---|---|---|---|---|---|---|

データの出典と注記:

- **日本%**: [16Personalities の日本カントリープロファイル](https://www.16personalities.com/country-profiles/japan)（回答者 79,290人）を出典とする分布。個別タイプの数値は同プロファイルを引用した複数の日本語まとめ（[MBTI診断ラボ](https://mbti-labo.jp/mbti-percentage/)、[パーソナルファイル](https://per-sonal.co.jp/personalfile/media/job-suitability-diagnosis/16personalities-japanese-percentage/)）でほぼ同一の値が確認できたものを小数1桁に丸めて採用（元値の合計≒100%）。
- **世界%**: 参照画像の値（一般に流通しているMBTI世界分布）。
- 日本はオンライン診断の自己選択サンプルであり厳密な国勢調査ではない。世界%とは調査母体が異なるため、**厳密な比較はできない「参考値」**である（懇親会のエンタメ用途として許容）。1位 INFP 16.4%・2位 ENFP 13.8% で、世界で最多の ISTJ/ISFJ が日本では少ないという傾向差自体が会話のネタになる。
- 日本のデータは全16タイプで合計約100%になるため、会場%との見た目の比較に使いやすい。ダッシュボードの説明文などで出典を示す必要はない（設計書に記録があれば足りる）。

### 動作仕様

- `<script>` 冒頭に `var CONFIG = { endpoint: "" };`（index.htmlと同じGAS URL）。
- 起動時に `new URLSearchParams(location.search).get("key")` でトークン取得。**無ければ**集計UIを出さず「このページの表示にはアクセスキーが必要です。幹事に確認してください。」の案内カードのみ表示（fetchもしない）。
- ポーリング: 起動直後に1回fetch → 以降 `setInterval` で**10秒ごと**に `GET <endpoint>?action=summary&key=<key>`。
  - 前回レスポンスの `counts` とJSON文字列比較し、**変化があったときだけ**再描画（レースの並び替え・タイル数値更新・合計/更新時刻の書き換え）。
  - fetch失敗・`ok:false` のとき: **前回データの表示を保持**し、ヘッダーに「⚠ 更新できません」を表示。次回成功時に自動で消す。`unauthorized` の場合は「アクセスキーが正しくありません」を表示しポーリングを止める。
  - 多重リクエスト防止: 前のfetchが完了していなければその回はスキップ。

## 17. 動作確認観点（拡張分）

- **エンドポイント未設定**: `CONFIG.endpoint` が空のとき、結果画面に送信ボタンが出ず従来通り動くこと。
- **送信フロー**: 診断完走→送信→ボタンが「送信しました ✓」になり、スプレッドシートに1行追加されること。
- **端末1回制限**: 送信後に「もう一回診断する」→結果画面のボタンが「送信しました ✓」のままであること。
- **冪等性**: localStorageを消して同じUUIDでPOSTしても行が増えないこと（`duplicate:true`）。
- **入力検証**: `curl` で 不正type / 不正uuid / 500文字超ボディ をPOSTし、すべて `ok:false` で拒否され行が増えないこと。
- **閲覧制御**: `key` なし・誤ったkeyのGETで `counts` が返らないこと。`dashboard.html` をkeyなしで開くと案内カードだけが出ること。
- **ポーリングと差分描画**: dashboard表示中に送信すると**10秒以内**に反映されること。データに変化がない間は再描画（アニメーション）が走らないこと。
- **棒グラフレース**: 回答が入ると棒が伸び、順位が変わると行が上下に**アニメーションしながら**並び替わること。同数タイプの順序が更新のたびに入れ替わらないこと。
- **障害時**: GASを一時的に落とす/ネットワーク断で、前回の集計表示が保持され「⚠ 更新できません」が出て、復旧後に自動で更新再開すること。
- **レイアウト**: 1100px以上（プロジェクタ）で4列、iPhone幅で2列になり横スクロールが出ないこと。
